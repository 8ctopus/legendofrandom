Re: Tutorial #13 is up

by random » Thu Jul 19, 2012 5:10 pm

mayorofarta, there are some binaries that Olly finds very challenging to keep track of breakpoints. Usually this occurs when there is code in a section that Olly thinks is not code (for example, a resource section). These sections can be overridden and made to be executable, but Olly doesn't know that. So he is wondering why the heck you would want to place a breakpoint on code that will never run (even though it will). This is one of the reasons you are losing your breakpoints.

Another reason is if the code happens to modify itself. If you place a breakpoint on an instruction, and then that instruction get's modified, Olly will lose the breakpoint.

Yet another reason for losing breakpoints is sometimes the binary has anti-breakpoint code in it that specifically traces through code and removes breakpoints or clears the hardware breakpoint registers.

All that being said, I would say that at least 10-15% of the time, Olly cannot keep track of breakpoints after a re-start. About the only thing you can do is save the breakpoints (using breakpoints manager plugin) after setting them and then reload them if you restart. Regarding hardware breakpoints, there isn't anything you can do except re-apply the BP after you restart the app. I know it's a pain.
